# SpicyLooper — Agent Context

## Project overview

A dark-themed HTML/JS audio loop player, usable as a browser app and as an Android APK (via Capacitor). Designed for musicians practicing over backing tracks. Single `index.html` — no build step for the web version.

## Stack

- **Web app**: vanilla HTML/JS, Web Audio API, JSZip (CDN), no framework
- **Android**: Capacitor 6, appId `com.spicylooper.app`
- **APK build**: Docker (two-stage: builder image + fast APK image)
- **Asset pipeline**: `@capacitor/assets` for icon generation

## File structure

```
index.html              — entire web app (single file)
img/
  stop.gif / play.gif / next.gif   — animated status GIFs (used by index.html as img/*)
  icon.png              — source app icon (1024×1024, used by Dockerfile.builder)
  app.jpg               — screenshot (used in README.md)
  pinout.png            — ESP32 Dev Board pinout diagram (used in README.md)
pedal/
  pedal.ino             — ESP32 BLE keyboard firmware (HijelHID_BLEKeyboard 0.5.0)
TakeFive/               — example project directory (intro/a/b/end .mp3)
TakeFive.spl            — same project packaged as .spl (zip renamed)
make_spl.sh             — packages a directory into a .spl file
Dockerfile.builder      — slow image: Java + Android SDK + Node + pre-warmed Gradle
Dockerfile.apk          — fast image: FROM spicylooper-builder, copies assets, builds APK
build-builder.sh        — run ONCE to build the builder image (~15 min)
build-apk.sh            — run per release; requires spicylooper-builder image
capacitor.config.json   — Capacitor config (appId, webDir: "www")
package.json            — Capacitor + @capacitor/assets deps
doc/TODO.md             — future work notes (PWA, Capacitor paths)
README.md               — end-user docs (APK install, usage, build, pedal wiring)
```

## Architecture

### Playback state machine

Six sections: `INTRO`, `A`, `B`, `C`, `D`, `END`.

- **INTRO**: plays once → auto-jumps to A (A shows as `next` immediately when INTRO starts)
- **A/B/C/D**: loop indefinitely; on click while another is playing → queued as `next`; loop finishes before switching
- **END**: plays once → stops
- `currentSec` / `nextSec` vars drive all UI state; `onended` callback on each `AudioBufferSourceNode` handles transitions

Key invariant: `currentSource.onended = null` is always cleared before calling `.stop()` to prevent ghost transitions.

### .spl format

A `.zip` file renamed to `.spl`. Contents: `intro.mp3`, `a.mp3`, `b.mp3`, `c.mp3`, `d.mp3`, `end.mp3`, optional `cover.jpg`, optional `project.json`. Files auto-detected by name prefix if `project.json` absent.

### APK Docker pipeline

```
build-builder.sh  →  spicylooper-builder image
                       (Java 17, Android SDK 34, Node 20, npm deps, Gradle warmed)
build-apk.sh      →  spicylooper-apk image  (FROM spicylooper-builder)
                       copies www/ assets, runs cap copy, generates icons, gradlew
                  →  output/SpicyLooper-debug.apk
```

Re-run `build-builder.sh` only when `package.json` or `capacitor.config.json` change.

### Capacitor www layout

`Dockerfile.apk` copies GIFs into `www/img/` (not `www/`) so that `index.html`'s `img/stop.gif` paths resolve correctly inside the APK WebView.

- [DECISION] Used `@capacitor/assets` for icon generation instead of ImageMagick.
  - Rationale: Node already present in container; `@capacitor/assets` is the Capacitor-native tool and handles adaptive icons automatically.
  - Date: 2026-06-06

- [DECISION] Two-stage Docker build (builder image + APK image).
  - Rationale: Android SDK + Gradle first-run takes ~15 min; caching in a persistent builder image makes subsequent APK builds fast.
  - Date: 2026-06-06

- [DECISION] `color-scheme: dark` meta tag + `-webkit-appearance: none` on buttons.
  - Rationale: Android WebView Force Dark Mode was mangling the green-on-black palette. These two CSS directives suppress it.
  - Date: 2026-06-06

- [DECISION] Manual loop re-triggering (no `source.loop = true`).
  - Rationale: `loop=true` gives no per-loop callback, so section switching would have no clean boundary. Manual re-trigger in `onended` gives precise control.
  - Date: 2026-06-06

### Key bindings (web app + pedal)

| Key | Pedal pin | Action |
|-----|-----------|--------|
| `0` / `Space` | 13 | INTRO |
| `1` / `]` | 12 | A |
| `2` / `=` | 14 | B |
| `3` / `-` | 27 | C |
| `4` / `[` | 26 | D |
| `9` / `.` | 25 | END |
| `,` | 33 | Stop (Panic) |

Pedal sends HID key codes via Bluetooth (pairs as keyboard). Letter aliases (a/b/c/d/x) removed.

### Known limitations

- `webkitdirectory` not supported in Android file picker → `multiple` attribute used as fallback; user must select individual files on Android instead of picking a folder
- JSZip loaded from CDN in browser; bundled via `wget` in Docker for APK offline use
- Debug APK only (not release-signed)

## Open issues / discoveries

- [OPEN] APK is debug-signed — cannot be uploaded to Play Store as-is.
  - Context: would need release keystore + `./gradlew assembleRelease` + signing config in `android/app/build.gradle`
  - Impact: blocks Play Store distribution; side-loading works fine

- [OPEN] Low-memory tablet issues — root cause not fully confirmed.
  - Context: `decodeAudioData` expands compressed MP3 to raw PCM float32 in RAM (~5 MB/min at 44.1 kHz stereo). Reverted the 22050 Hz mono downsample fix; to re-enable, restore `downsample()` function and call it after `decodeAudioData` in both `loadSplFile` and `loadDirectory`.
  - Impact: may cause tab crash on low-RAM devices with long audio files

- [OPEN] PWA / GitHub Pages path not pursued yet — documented in `doc/TODO.md`.
